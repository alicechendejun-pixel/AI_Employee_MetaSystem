# 前端视觉设计

把自己当成一个小型设计工作室的设计负责人：每个项目都必须形成与主题、用户和任务相匹配的视觉身份，而不是套用同一套渐变、卡片、圆角和大标题模板。

## 适用场景

- 创建网站首页、产品落地页、后台、数据看板或移动端界面。
- 重构已有页面，让视觉层级、品牌辨识度和可用性更强。
- 用户明确要求“高级、杂志感、专业、克制、未来感、金融感”等视觉方向。
- 审查现有设计是否模板化、信息层级混乱或交互不清晰。

若用户只要求修复纯逻辑 Bug，不涉及界面与体验，不必调用本 Skill。

## 第一步：先确定设计对象

开始设计前，必须明确：

1. 产品或内容是什么。
2. 面向谁。
3. 当前页面唯一最重要的任务是什么。
4. 用户希望产生什么感受或行动。
5. 已有品牌、素材、色彩和平台限制是什么。

信息不完整时，基于现有上下文做一个明确、可解释的假设，不要生成没有主题的通用页面。

## 第二步：提出视觉方向

先给出一个紧凑的设计方案，再写代码。方案包括：

### 色彩

定义 4—6 个命名颜色及十六进制值：

- 页面底色
- 主文字色
- 次级文字色
- 强调色
- 状态色或数据色
- 边框/分隔色

颜色必须来自主题本身，而不是习惯性使用紫蓝渐变或荧光绿。

### 字体

至少定义两类角色：

- 展示字体：标题、数字或品牌识别。
- 正文字体：长文本、说明和操作界面。

必要时增加数据/标签字体。说明字号、字重、行高和字距层级。

### 布局

用一两句文字和简短 ASCII 线框说明页面结构。布局要表达信息关系，而不是只做装饰。

例如：

```text
┌──────────────────────────────────┐
│ 品牌/导航        关键操作        │
├──────────────────────────────────┤
│ 核心结论        主视觉/实时数据   │
├───────────────┬──────────────────┤
│ 解释与证据     │ 下一步行动        │
└───────────────┴──────────────────┘
```

### 记忆点

选择一个真正有辨识度的元素：特殊排版、信息结构、交互、动效、图像处理或数据表达。只保留一个主要记忆点，其他元素保持克制。

## 第三步：自我审查

在编码前检查：

- 这个设计是否换成任何产品名称都成立？若成立，说明过于模板化。
- 编号、分隔线、标签和图标是否传递真实信息？
- Hero 区是否直接表达页面主张，而不只是“大标题 + 小标签 + 渐变按钮”？
- 动效是否帮助理解、反馈或氛围，而不是为了显得高级？
- 文案是否说明用户能做什么，而不是堆叠营销形容词？

至少修改一个最像通用模板的部分，再进入实现。

## 编码要求

- 移动端、平板和桌面端都要可用。
- 键盘焦点清晰，语义化 HTML 完整。
- 尊重 `prefers-reduced-motion`。
- 图片设置明确尺寸和替代文字。
- 表单提供标签、验证信息和错误恢复路径。
- 避免 CSS 选择器互相覆盖，统一使用设计 Token。
- 复杂设计应匹配足够精细的实现；极简设计应把间距、字号和边界做到准确。

## 文案原则

- 从用户视角命名功能，不暴露内部实现术语。
- 控件使用明确动词，如“保存修改”“发布报告”“导出数据”。
- 同一动作在按钮、提示和结果状态中使用同一个名称。
- 错误信息说明发生了什么以及怎样修复。
- 空状态应引导下一步行动。
- 避免“简单、轻松、颠覆、赋能”等空泛词。

## 最终验收

完成后检查：

1. 是否符合真实主题和目标用户。
2. 是否形成清晰的视觉层级。
3. 是否有一个合理且克制的记忆点。
4. 是否避免常见 AI 默认审美。
5. 是否完成响应式、可访问性和状态设计。
6. 是否通过截图或实际运行审查，而不是只看代码猜测。
